# VERSIONING_AND_COMPOST_POLICY.md
## Version 1.0
## Status: Active
## Scope: All Directories
## Location: /core

---

# PURPOSE

This document defines:

• How versions are numbered  
• When documents are moved to /compostandgrowth  
• What counts as active  
• How historical integrity is preserved  

Flow evolves.
Nothing is deleted.
Old structure becomes growth medium.

---

# I. VERSION NUMBERING STANDARD

All active documents must use the following format:

MAJOR.MINOR

Examples:
1.0
1.1
2.0
3.2

---

## A. MAJOR VERSION (X.0)

Increase MAJOR when:

• Structural logic changes  
• Governance thresholds change  
• Baseline values change  
• Core protocol direction changes  

A major version replaces the previous major version.

Previous major version must be moved to:

/compostandgrowth

---

## B. MINOR VERSION (X.Y)

Increase MINOR when:

• Clarifications are added  
• Language is refined  
• Non-structural corrections occur  
• Formatting changes  
• Interpretive precision improves  

Minor updates do NOT move previous versions to compost.

Only major shifts compost the previous root version.

---

# II. COMPOSTANDGROWTH RULE

Each directory must contain:

/compostandgrowth

When a MAJOR version is released:

1. The previous MAJOR file is moved to:
/compostandgrowth

2. Filename must include version tag:

Example:
RESOURCE_METRIC_STANDARDS_v2.0.md

3. Compost files are immutable.

They are historical record, not editable drafts.

---

# III. ACTIVE DOCUMENT IDENTIFICATION

Only one file per protocol may exist at root level.

Example:

/core/RESOURCE_METRIC_STANDARDS.md   ← active
/core/compostandgrowth/RESOURCE_METRIC_STANDARDS_v2.0.md

Root-level file is always the current authoritative version.

---

# IV. FORK VISIBILITY

If a fork occurs:

Fork must:

• Declare divergence version
• State parent version number
• Maintain compost structure

This ensures genealogical traceability.

---

# V. STRUCTURAL PRINCIPLE

Flow does not erase its past.

It composts it.

Growth requires memory.
